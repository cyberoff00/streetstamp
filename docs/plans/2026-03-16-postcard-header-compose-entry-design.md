# Postcard Header Compose Entry Design

日期：2026-03-16
状态：已评审通过（用户确认）

## 1. 目标

在明信片发件箱页面的 header 右侧新增一个信封 icon，作为“新建明信片”入口。

点击后复用现有 1v1 明信片发送页面，但不再要求从好友主页进入。用户会先在页面顶部通过“添加收件人”选择 1 位好友，然后继续原有的城市、照片、文案、预览和发送流程。

## 2. 范围

本次只做单发，不做群发。

包含：
- 明信片页面 header 右侧新增信封入口
- 从该入口进入 composer 时支持“未预填好友”状态
- composer 顶部新增/改造为“添加收件人”入口
- 只能选择 1 位好友
- 选完好友后复用现有单发逻辑

不包含：
- 多收件人
- 发件箱聚合展示
- 新的发送接口

## 3. 页面与交互

### 3.1 新入口

- 页面：`PostcardInboxView`
- 位置：header 右侧
- 图标：系统信封 icon

点击后进入现有 `PostcardComposerView`。

### 3.2 Composer 行为

入口分两种：

- 从好友主页进入：
  - 保持现有行为，默认带入该好友
- 从发件箱 header 进入：
  - 不预填好友
  - 顶部显示“添加收件人”
  - 用户从好友列表中选择 1 位好友

选中好友后，下方内容保持不变：
- 选择城市
- 上传照片
- 输入文案
- 预览
- 发送

## 4. 数据与架构

当前发送模型仍保持 1v1。

建议做法：
- 将 `PostcardComposerView` 的输入从“必须有 friendID/friendName”改为“可选预填好友”
- 新增一个轻量的收件人选择状态，仅保存当前选中的 1 位好友
- `PostcardPreviewView` 和 `PostcardCenter.createDraft` 继续使用单收件人参数

这样可以最小化改动，避免波及现有发送链路。

## 5. 约束

- 最多只能选 1 位好友
- 未选择好友时不能进入预览或发送
- 不允许选择自己
- 只从现有好友列表中选人，不新增搜索后端能力

## 6. 验收标准

1. 明信片页面 header 右侧出现信封按钮
2. 点击按钮可以进入明信片 composer
3. 从该入口进入时默认没有好友
4. 顶部可以添加 1 位收件人
5. 选择好友后可继续原有单发流程
6. 从好友主页进入时原有 1v1 行为不回归

## 7. 测试重点

- composer 在“预填好友”和“空收件人”两种模式下都能正常工作
- 未选好友时预览按钮不可用
- 从发件箱入口进入可以完成选择好友并发送
- 原有好友主页入口仍可正常发送
