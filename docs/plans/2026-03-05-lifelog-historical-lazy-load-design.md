# Lifelog 历史轨迹分层加载设计

日期：2026-03-05

## 背景
当前 `LifelogStore.load()` 在加载后会执行“仅保留今天 passive 轨迹”的裁剪，导致历史轨迹在重启/重装后看起来消失。用户期望保留历史，并降低首屏/切日卡顿风险。

## 目标
- 首屏优先加载近 7 天轨迹。
- 历史轨迹按天懒加载渲染（切到某天再取该天）。
- 后台构建“按天索引”以降低首次切换历史日期开销。
- 保持 mood 持久化行为不回退。

## 非目标
- 本阶段不改动磁盘文件格式为按天分片。
- 不引入额外数据库。

## 方案
1. 取消 load 后的“仅今天裁剪”逻辑，保留全量 points 作为历史真源。
2. 引入“近 7 天窗口坐标”作为 `coordinates`（用于默认视图与近景路径）。
3. 保留现有 `mapPolyline(day:)` 与 `mapPolylineViewport(day:)` 的按天 API，历史日期通过 `coordsFor(day:)` 懒取。
4. 在 `load()` 后异步（utility）构建 day -> coords 索引；构建完成后回填 `dayCoordsCache`，后续按天取数 O(1) 命中。
5. 优化 `dayKey` 计算，移除高频 `DateFormatter` 实例化。

## 风险与控制
- 风险：首次后台构建索引期间，首次切某些历史天仍会走一次扫描。
  - 控制：保留现有扫描 fallback，功能正确优先；索引完成后自动提速。
- 风险：`coordinates` 改为近 7 天后影响依赖该字段的测试。
  - 控制：补充/更新测试，显式验证“默认近 7 天 + 按天历史可查”。

## 验收标准
- 重启后昨天及更早轨迹不再被清空。
- 默认非指定日期视图只显示近 7 天。
- 点选历史日期能渲染当天轨迹。
- 相关单测通过。
